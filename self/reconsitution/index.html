# 想法

# 重构之心

营地应该要比你来时，更干净。

身处开发团队中，无论是新人还是老手我认为重构之心应该一直保持。当然也不是呼吁大家天天重构，重构虽然是一种好习惯，但也是因地制宜。我们为什么需要重构代码？是提升技术能力还是加快代码书写速度？还是单纯只是因为这份代码不是出自你手？觉得别人的代码充斥着腐败不堪的味道？以上重重问题可能驱使着你重构现有代码，但是记住不要盲目给自己提供兴奋剂。通俗点说不应该盲目的去重构，有时候太过于“勤劳”也是原罪。

# 护林者

辛勤劳作的人当然是值得称赞和鼓励的，但并非任何时候你所做的都能被人称赞和嘉奖。可能有时候都是徒劳，当你正在维护一个私有项目的时候，无论是什么语种、无论大小、无论复杂简单，此时此刻你就是这片“领地”的“护林者”。换而言之此时此刻应该没有人更比你懂这片“森林”，无论是直径小路、还是艰险泥泞，你都应该是了如指掌。当你需要肩负“护林者”职责的时候，就应该去这片林子里将那些图谋不轨的“人”赶出去，将那些阻挡了上山人去路的灌木移开，将那些容易伤人肌肤的荆棘劈断。提前预警“森林”中的黑暗境地，让前来这里冒险的人小心行事。

# 安慰注释的陷阱

在《代码简洁之道》这本书中提到过注释也是一种让代码不干净的元素之一，但也说过必要的注释也是应该的。其实并不矛盾，当你为你的代码书写注释时，我相信大部分情况肯定是为了自己未来读这一块，能不迷路。即使还是会迷路，总比光秃秃的要好吧，你在内心这么安慰自己。诚然我自己也是如此，但是这样就陷入了“安慰注释”的陷阱。你认为无论自己的代码如何书写你都不重要，你认为一行注释在未来可以为你指路，甚至可以帮助到冒险家。但老兄你知道吗，“铺平道路不放路牌”和“草草定钉那几块破板子往哪一甩，然后假惺惺的在一个破牌子上写上一段温馨提示：此处通往 xxx，祝您顺利”是两种处理事情的态度和两种截然不同的职业素养体现。我相信你作为软件工程师，你一定是希望自己是后者，如果你不在意这些细节可以无需往下去阅读，因为你的傲慢和不负责会让你未来尝尽苦头，并且被人在背后不断抱怨你留下来的烂摊子。

# 请做好指路牌

试想一下，你在写下你业务代码第一行时。是不是等同于留下了“线索”，我将我自己编写的代码大到类命名，小到变量命名我都称之为线索。因为后面的冒险者会寻迹而来。如果你做好了整体的道路规划和良好的注释习惯，我相信冒险家们一定会感叹“真是不错的设计！”，那么何为不错的设计？这个问题没有统一答案，但是能够确定的是语义化和共鸣性无论用在变量名还是方法名他都是非常不错的做法，这能够让人阅读时不那么晕头转向。可以夸张的说如果做好了以上工作，即使你不写一行注释冒险家也能闻着“花香”到达终点，但为了让整个阅读流程变得轻松愉悦，适当制作好指路牌。也是“可选”加分项，好的注释应该是简洁明了的，而不是罗里吧嗦像个唐僧不断念经。
